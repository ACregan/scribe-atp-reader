Collapse the duplicate Author Tree route

app/routes/tree.tsx · app/routes/tree-alt.tsx · app/routes.ts

Problem. tree.tsx and tree-alt.tsx are byte-identical except for a type import and a function name. Both render the Author Tree module at two seams — /:author and /:author/site.standard.publication — but nothing varies across them. One adapter means a hypothetical seam; here there's only one real implementation wearing two files.

Solution. Point both route entries in routes.ts at the single file routes/tree.tsx, giving the second entry an explicit id (React Router derives route ids from the file path, so a bare second reference collides). Delete tree-alt.tsx. Confirmed with npm run typecheck — passes clean, one generated +types/tree.ts serves both routes.

Deepen Site loading into one module

app/routes/site.tsx · app/routes/group.tsx · app/routes/article-site.tsx · (prior art: app/lib/loadAuthorTree.server.ts)

Problem. Three loaders each hand-roll the same call: withNotFound(() => fetchSite(author, \`https://${'$'}{siteDomain}\`, signal)). The URL-building convention (bare domain → HTTPS URL) and the 404-on-missing-record decision are knowledge, not just code — repeated three times, they can drift independently. This is exactly the shallow-wrapper smell loadAuthorTree.server.ts was already extracted to avoid for the tree routes; site-fetching never got the same treatment.

Solution. Add loadSite.server.ts exporting loadSite(author, siteDomain, signal), mirroring loadAuthorTree's shape exactly. The three routes call it and keep their own group/article lookups on top.

Pull the pure logic out of ArticleView and ArticleItem

app/components/ArticleView.tsx · app/components/ArticleItem/ArticleItem.tsx

Problem. Two non-trivial pure decisions live inside JSX components with no tests: bskyPostUrl() parses an AT-URI into a bsky.app link by array index, and the reading-time calculation applies a 225-wpm constant — both in ArticleView.tsx. ArticleItem separately branches on siteDomain/groupSlug presence to pick a link target. Every other piece of decision logic in this repo — resolveAuthorPath, errorBoundaryContent, withNotFound, tagArticles, publishedOn — was deliberately extracted to lib/*.ts specifically so it's "testable without a DOM environment," per their own comments. These two are the exceptions.

Solution. Extract bskyPostUrl and a reading-time function into lib/articleMeta.ts; extract the href branch out of ArticleItem into lib/buildArticleHref.ts. Each gets a .test.ts sibling in the same style as publishedOn.test.ts.

Unify the two publishedOn search functions

app/lib/publishedOn.ts

Problem. findPublishedOn and findPublishedOnInGroup share the same canonical-URL building and the same article-in-group search; they differ only in scope — all sites vs. one named group on one site. Two names for one search, at two call sites (article.tsx, article-site.tsx).

Solution. One function, findPublishedOn(sites, articleRkey, { groupSlug? }), with the group-scoped route passing a single-site array and a slug. Keep both call sites simple to read.

  • Leverage: one search implementation instead of two aligned-by-hand copies
  • Risk: an optional-scope parameter can itself drift toward shallow if it grows further — worth a second look before committing
Speculative

ArticleItem's prop bag is edging toward shallow

app/components/ArticleItem/ArticleItem.tsx

Problem. Five props — state, siteDomain, groupSlug, showDescription — serve two distinct call-site modes (a nested tree item under Group/Site, and a flat description-list item with a state badge), and siteDomain/groupSlug are only meaningful together. The interface is starting to enumerate its callers instead of hiding them.

Solution. Pair with candidate 3: once href-building moves to buildArticleHref, callers could pass a resolved to string directly, dropping siteDomain/groupSlug from the component's interface entirely.

  • At 59 lines the component is still small — splitting now may cost more locality than it buys
  • Re-examine after candidate 3 lands; it may resolve itself

Top recommendation

Pull the pure logic out of ArticleView and ArticleItem

This repo has already validated the deepening move five times over — extract the decision, test it without a DOM, keep the component to rendering. ArticleView and ArticleItem are the two places that move hasn't reached yet. Doing it here costs nothing new to learn and closes the last gap in an already-consistent architecture. Start there; candidate 1 (route dedup, already verified safe) is worth taking in the same pass since it's a pure deletion.
